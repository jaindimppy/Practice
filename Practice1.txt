I— Interface Segregation Principle (ISP)
One big interface forces classes to implement methods they don’t need.

public interface IWorker
{
    void Work();
    void Eat();
}

Now:
public class HumanWorker : IWorker
{
    public void Work() { }
    public void Eat() { }
}

public class RobotWorker : IWorker
{
    public void Work() { }
    public void Eat() { } // ❌ Robot does NOT eat but is forced to implement
}

👉 Problem:
Robot does not eat, but interface forces it.

✅ Good Example (Follows ISP)
Split into small, focused interfaces.

public interface IWork
{
    void Work();
}

public interface IEat
{
    void Eat();
}

Now implement only what you need:
public class HumanWorker : IWork, IEat
{
    public void Work() { }
    public void Eat() { }
}

public class RobotWorker : IWork
{
    public void Work() { }
    // No Eat() required ✔
}

👉 Benefits:
Classes implement only what they need.


D — Dependency Inversion Principle (DIP)

High-level class depends on low-level class.
High-level classes should not depend on low-level classes.
Both should depend on abstractions (interfaces).

❌ Without DIP (bad)
public class NotificationService
{
    private readonly EmailService _emailService = new EmailService();

    public void Send(string message)
    {
        _emailService.SendEmail(message);
    }
}
✅ With DIP (good)

High-level class depends on an abstraction, not a concrete class.

public interface INotificationSender
{
    void Send(string message);
}

public class EmailSender : INotificationSender
{
    public void Send(string message) => Console.WriteLine("Email: " + message);
}

public class SmsSender : INotificationSender
{
    public void Send(string message) => Console.WriteLine("SMS: " + message);
}

public class NotificationService
{
    private readonly INotificationSender _sender;

    public NotificationService(INotificationSender sender)
    {
        _sender = sender;
    }

    public void Notify(string msg)
    {
        _sender.Send(msg);
    }
}

Benefits:
You can add WhatsApp, Push, Teams, Slack without touching NotificationService.
Testable.
Extensible.

Non-Generic Collection (ArrayList)
Before generics, collections like ArrayList stored elements as object.
That means value types (like int) must be boxed when added, and unboxed when retrieved.
using System;
using System.Collections;

class Program {
    static void Main() {
        ArrayList list = new ArrayList();

        int num = 42;
        list.Add(num);   // Boxing happens here (int → object)

        int retrieved = (int)list[0]; // Unboxing (object → int)
        Console.WriteLine(retrieved);
    }
}
⚠️ Problems:
- Performance overhead: Boxing allocates memory on the heap, and unboxing requires type casting.
- Type safety issues: If you accidentally cast to the wrong type, you’ll get a runtime error.

✅ Generic Collection (List<int>)
With generics, the collection is strongly typed.
No boxing/unboxing occurs because the compiler knows the type (int in this case).
using System;
using System.Collections.Generic;

class Program {
    static void Main() {
        List<int> list = new List<int>();

        int num = 42;
        list.Add(num);   // No boxing (stored directly as int)

        int retrieved = list[0]; // No unboxing (already int)
        Console.WriteLine(retrieved);
    }
}
🎯 Benefits:
- No boxing/unboxing overhead → faster performance.
- Type safety → compiler prevents invalid casts.
- Cleaner code → no need for explicit casting.
