Using versioning

Database: Add a version column (integer).

Initial State: Balance = $100, Version = 1.

Step 1: User A reads the row (Balance: $100, Version: 1).

Step 2: User B reads the row (Balance: $100, Version: 1).

Step 3: User A calculates $150 and sends: UPDATE account SET balance=150, version=2 WHERE id=1 AND version=1.



Result: Success! The database sees version 1 matches, updates the row, and increments it to 2.

Step 4: User B calculates $130 and sends: UPDATE account SET balance=130, version=2 WHERE id=1 AND version=1.



Result: Fail! The database sees that the version is now 2, not 1. The WHERE clause fails.

Next Step: Your API should catch this failure, re-read the new balance ($150), ask the user if they still want to add $30, or automatically retry the calculation.

2)Difference between Encapsulation and Abstraction
1. Encapsulation (Protecting the "Inside")
You are 100% correct: Encapsulation is about access control. By making variables private, you prevent the "outside world" from breaking your object's internal state.
Key Tool: Access Modifiers (private, protected).
The "Hiding" Goal: To prevent corruption (e.g., someone setting age = -5 or balance = -999999).
The "Getter/Setter" Bridge: These aren't just for reading/writing; they act as Gatekeepers. They are where you put your validation logic (if (age > 0) { ... }).

2. Abstraction (Simplifying the "Outside")
Abstraction is about hiding implementation complexity.
Key Tool: Interfaces, Abstract Classes, and Public Methods.
The "Hiding" Goal: To prevent cognitive overload. If I am using your Bank class, I don't want to see the 50 lines of code you wrote to verify a user's identity, calculate interest, and update a database. I just want to see the transfer() method.

6)Multiple Inheritance
which occurs when a class inherits from two classes that both have a method with the same name, leading to ambiguity about which method the child class should use.
Imagine you have two interfaces, IWork and IExercise, and both have a method called Start().
public interface IWork
{
    void Start();
}
public interface IExercise
{
    void Start();
}
If a class implements both, C# doesn't know which Start() is which. To solve this, you prefix the method name with the Interface name.
public class Manager : IWork, IExercise
{
    // This handles IWork.Start()
    void IWork.Start() 
    {Console.WriteLine("Opening Laptop and starting work...");
    }
    // This handles IExercise.Start()
    void IExercise.Start() 
    {
        Console.WriteLine("Hitting the gym and starting exercise...");
    }
}

How to call them
Because the methods are "explicit," they are hidden from the class's normal public view. You must cast the object to the specific interface you want to use.
Manager myManager = new Manager();
// myManager.Start(); // This would cause a COMPILE ERROR (Ambiguous)

// Correct Way:
IWork worker = myManager;
worker.Start(); // Outputs: Opening Laptop and starting work...

IExercise athlete = myManager;
athlete.Start(); // Outputs: Hitting the gym and starting exercise...

ACID
Atomicity 
It ensures that a transaction is treated as a single, indivisible unit. Either the entire transaction succeeds, or the entire transaction fails. There is no "half-finished" state.
Consistency
 If a database column has a rule that "Balance cannot be negative," and a transaction tries to withdraw more money than is available, the database will reject the transaction to maintain Consistency

Isolation 
Each transaction should feel like it is the only one running.
If two people try to buy the last concert ticket at the exact same millisecond, Isolation ensures the system processes one first and tells the second person the ticket is sold out, rather than selling the same ticket twice.

Durability 
Durability guarantees that once a transaction has been committed, it will remain committed even in the case of a system failure

What is Caching?
Caching is the process of storing copies of frequently requested data in a temporary, ultra-fast storage layer (usually RAM) so that future requests for that data can be served much quicker than fetching it from the original source (like a slow database or an external API).

In-Memory Cache (IMemoryCache)
The data is stored directly in the RAM of the web server hosting your application.
Pros: Blazing fast because there is zero network overhead.
Cons: If your app restarts, the cache is wiped out. More importantly, if you host your API on multiple servers (a server farm), Server A won't know what Server B has cached, leading to inconsistent data.

Distributed Cache (IDistributedCache)
The cache is stored in an external, independent service that all your web servers connect to.
Pros: If your application servers restart, the cache survives. Every web server looks at the exact same cache pool, ensuring data consistency across your entire infrastructure.
Cons: Slightly slower than in-memory because data has to travel over the network, and objects must be serialized to bytes/strings.

Redis Cache
Redis is the industry-standard software used to power a Distributed Cache. It is an open-source, in-memory, key-value data store. .NET provides native support to plug Redis directly into the IDistributedCache interface with a single line of setup code in Program.cs.
