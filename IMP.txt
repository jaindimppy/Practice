var student = context.Students.FirstOrDefault(s => s.Id == 1);
student.Name = "John Updated"; // change the name

context.SaveChanges(); // EF sees the change and updates the database
EF tracks the original name and sees it changed to "John Updated", so it executes:
UPDATE Students SET Name = 'John Updated' WHERE Id = 1


🚫 Without Tracking (using AsNoTracking()):

var student = context.Students.AsNoTracking().FirstOrDefault(s => s.Id == 1);
student.Name = "John Updated";

context.SaveChanges(); // ❌ Nothing happens
Since AsNoTracking() was used, EF is not tracking changes to student, so calling SaveChanges() does nothing.

=>Eagar Loading
Loads related data at the same time as the main entity using .Include().
var students = context.Students
                      .Include(s => s.Courses)
                      .ToList();
This loads all students and their related courses in a single query.
Use when you always need the related data immediately.

=>Eplicit Loading
Loads related data manually after the main entity is loaded, using .Entry().Collection().Load() or .Reference().Load().
var student = context.Students.FirstOrDefault(s => s.Id == 1);

// Explicitly load related courses
context.Entry(student)
       .Collection(s => s.Courses)
       .Load();
This runs two separate queries:
One for Student
Another for Courses of that student

Use when you don’t always need the related data but decide later at runtime.

=>Lazy Loading
 Loads related data automatically when you access the navigation property — only if it's not already loaded.
var student = context.Students.FirstOrDefault(s => s.Id == 1);
var courses = student.Courses; // Triggers separate query if not loaded

For this to work:
1. Navigation properties must be virtual
2. Enable Lazy Loading Proxies in Startup/Program
services.AddDbContext<AppDbContext>(options =>
    options.UseLazyLoadingProxies()
           .UseSqlServer(Configuration.GetConnectionString("Default")));
3. dotnet add package Microsoft.EntityFrameworkCore.Proxies
Use when you may or may not need related data, and want it fetched only if accessed.
proxy is needed only in lazy loading
problem while using lazy loading:
N+1 problem
var users = _context.Users.ToList(); // 1 query: SELECT * FROM Users

foreach(var user in users)
{
    var events = user.Events; // triggers N queries (one per user)
}

So total queries = 1 + N → this is the N+1 problem.

=>Change tracking:
Change tracking is the mechanism Entity Framework (EF) uses to monitor changes to entities and their relationships so it knows what to update when SaveChanges() is called.

It means EF tracks changes such as:
Adding/removing items in a collection (e.g., student.Courses.Add(course))
Assigning a related entity (e.g., student.Advisor = someTeacher)
Breaking relationships (e.g., student.Courses.Remove(course))

public class Student
{
    public int Id { get; set; }
    public string Name { get; set; }

    // Navigation property must be virtual for lazy loading
    public virtual ICollection<Course> Courses { get; set; }
}

public class Course
{
    public int Id { get; set; }
    public string Title { get; set; }

    public int StudentId { get; set; }
    public virtual Student Student { get; set; }
}

Ex:
var student = db.Students.Include(s => s.Courses).First();
var newCourse = db.Courses.First(c => c.Id == 5);
student.Courses.Add(newCourse); // EF tracks this addition
db.SaveChanges(); // EF generates a linking INSERT in the join table

EF tracked the change (Add) and knows to update the relationship table.
like this:
| StudentId | CourseId |
| --------- | -------- |
| 1         | 5        |

EF won’t track changes if:
=Proxy creation is disabled.
=Navigation properties are not virtual (EF can't override them).
Ex.2

In EF Core, change tracking means EF monitors entities you load from the database and tracks changes so it can automatically generate the right SQL UPDATE/DELETE commands when you call SaveChanges().

public class User
{
    public int Id { get; set; }
    public string Name { get; set; }

    public virtual ICollection<Event> Events { get; set; }
}

var user = await context.Users.FirstAsync();   // Only user is loaded here
user.Name = "New Name";                        // EF tracks this ✅

user.Events.First().Title = "New Event Title"; // ❌ Not tracked if Events were not loaded

👉 But if lazy loading is enabled, user.Events.First() will load Events, and now EF can track changes to event entities too.
1. user.Events.First()
Since Events is a virtual navigation property, and EF Core's lazy loading proxy is active,
EF automatically loads Events from the database when you access .Events.
2. Title = "New Event Title"
EF adds this change to the Change Tracker.
3. await context.SaveChangesAsync()
EF generates an UPDATE SQL command for the Event table to save the changed title.

💡 Rule of Thumb:

Singleton → “once per app.”
Lifetime: A single instance is created once and shared across the entire application.

Scoped → “once per request.”
Lifetime: A new instance is created per HTTP request (or scope). Within that request, all usages get the same instance.

Transient → “new every time.”
Lifetime: A new instance is created every time the service is requested.
